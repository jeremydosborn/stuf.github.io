# Funding

Funding helps turn STUF from an early technical prototype into reviewed, documented infrastructure that embedded and fleet teams can actually rely on.

## Why this matters

Secure update infrastructure is foundational. A device that accepts an unverified firmware image is a device that can be turned against its operator. The problem is not exotic — compromised update channels have caused real incidents across industrial, automotive, and cloud systems.

STUF's goal is to make the verification layer small enough to audit and deploy on constrained hardware. Getting there requires sustained time: for security review, for working with pilot partners on real constraints, and for hardening the specification.

## Funding priorities

These are listed in order of impact, not in order of cost.

### 1. Security review

The most important use of funding. The trust kernel needs independent scrutiny before it can be recommended for production use. This means paying a qualified reviewer for a focused engagement on the verification logic, encoding, and error handling.

### 2. Embedded verification profile

Working with real embedded targets — not just simulated constraints — to verify that the no-heap profile holds under production conditions. This requires hardware, time, and iteration with engineers who run constrained systems.

### 3. TUF profile hardening

The TUF-oriented verification path needs closer alignment with the TUF specification and compatibility testing against existing TUF infrastructure.

### 4. Pilot support

Helping pilot partners instrument, test, and report on STUF in real update pipelines. Pilots surface assumptions that review alone cannot find.

### 5. Documentation and examples

Clear, complete documentation lowers the barrier to review and adoption. Better examples make the security model legible to implementors.

### 6. Interoperability fixtures

A shared fixture set for testing compatibility across implementations. Useful for anyone building on or against the STUF specification.
